KEY DESIGN DECISIONS (never change these)
═══════════════════════════════════════════════════════
- BigDecimal for money, never double
- No setters on domain objects
- Entity/Domain/DTO separation
- H2 for tests, PostgreSQL for dev
- Flyway owns schema, never edit existing migrations
- JWT stateless auth
- TDD always: red → green → refactor
- Pain first: feel the problem before the solution
- AWS for cloud (Phase 11)
- Spring Authorization Server for OAuth 2.0 (Phase 9)
- Guard clauses pattern (fail fast, happy path last)
- IAccountService interface deferred to Phase 9
  (introduce when second implementation exists)
- No SimpleAsyncTaskExecutor ever
  (always ThreadPoolTaskExecutor)
- Spring Batch over @Scheduled for critical batch jobs
- Redis over in-memory cache for microservices

INTERVIEW PREP NOTES
═══════════════════════════════════════════════════════
- Answer structure: WHAT + HOW + WHY
- JWT: explain full flow from login to protected endpoint
- Redis: mention "shared cache layer across all instances"
- Optimistic locking: mention WHERE version=? SQL,
  OptimisticLockException, 409 response
- Spring Batch vs Kafka: batch=bulk processing,
  Kafka=real-time events, neither replaces the other
- Interface vs class: Dependency Inversion,
  introduce interface when second impl exists
- Practice: explain each concept out loud daily
  without notes
